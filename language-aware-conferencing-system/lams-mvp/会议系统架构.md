下面我把你文中的 **SAR 按 ASR/语音识别链路理解**；如果你说的 SAR 是 speaker-aware recognition、source audio separation 或会议声源分离，我也把它放到“前处理 + 说话人分离/识别”层里一起设计。

## 结论：不要照抄 Meetily，要升级成“本地会议 AI 流水线平台”

你给的 Meetily 文章最值得借鉴的不是某个模型，而是三个工程原则：**本地优先、音频链路工程化、实时链路与离线链路分离**。文章里提到 Meetily 用 Tauri/Rust 把前端、音频引擎、转写引擎、摘要引擎和 SQLite 封装成单体本地应用，并通过类型安全 IPC 调用后端；音频链路则做了采集、重采样、VAD、响度归一化、降噪、混音、编码等处理；实时转写还用单 worker 保证 chunk 顺序，避免实时字幕乱序。   

但你要做的不是单纯“会议纪要工具”，而是 **本地会议 + ASR/SAR + T2T 翻译 + TTS 语音回放 + 可训练 + 可插拔各阶段**。所以更好的方案是：**把 Meetily 的本地桌面工程能力保留，但把模型推理层改成插件化 DAG/流水线平台**。核心不要写死 Whisper/Parakeet/Ollama/NeuTTS，而是让每个阶段都遵守统一事件协议，能替换、微调、回放、A/B 测试、量化部署、回滚。

## 一、会议系统架构：分成两种部署模式

### 模式 A：边车型本地会议助手，适合接入 Zoom / Teams / Google Meet / 腾讯会议

这是最快落地的方案。应用不自己做视频会议，只做本地 AI 旁路：

```text
系统音频/麦克风捕获
        ↓
音频前处理：重采样、VAD、降噪、AEC、AGC、混音
        ↓
ASR / SAR：语音识别 + 说话人分离/识别
        ↓
T2T：文本翻译、术语表、风格改写
        ↓
TTS：目标语言语音合成
        ↓
字幕面板 / 会议摘要 / 虚拟麦克风输出 / 本地录音归档
```

优点是落地快、风险低、兼容现有会议软件。缺点是你对会议媒体层控制较弱，尤其是把 TTS 回灌到会议时，需要虚拟麦克风、回声消除和防自激机制。

### 模式 B：自研/自托管会议系统，适合企业内网会议平台

如果你要搭完整会议系统，应优先用 **WebRTC SFU**，不要用 MCU 混流架构。SFU 的职责是转发媒体流，不做重编码或混音，延迟和扩展性更好；mediasoup 文档明确说 SFU 接收端点流并转发给其他端点，接收端可以选择需要的空间/时间层，相比 MCU 性能更好、吞吐更高、延迟更低。([Mediasoup][1]) LiveKit 也是基于 SFU，把它描述成低延迟、高带宽媒体转发路由器；Jitsi 也提供自托管会议方案和 Videobridge/SFU 路线。([LiveKit ドキュメント][2])

推荐架构：

```text
Web / Desktop / Mobile Client
        ↓ WebRTC
SFU：LiveKit / Jitsi / mediasoup
        ↓ 订阅每个音频 track
AI Bot / AI Worker
        ↓
ASR / SAR → T2T → TTS
        ↓
DataChannel 字幕 + Bot 音频 track + 会议纪要
```

关键点是：**AI 不要塞进 SFU 的核心媒体转发路径**。AI 应该作为一个“机器人参会者”或“旁路媒体订阅者”订阅音频 track，输出字幕和 TTS track。这样 SFU 仍然保持低延迟和稳定，AI 模型崩溃不会拖垮会议。

## 二、总体概要设计

### 1. 核心模块

| 模块              | 职责                       | 推荐实现                                                             |
| --------------- | ------------------------ | ---------------------------------------------------------------- |
| Meeting Core    | 房间、成员、音视频轨道、权限、录制        | 自研用 LiveKit/Jitsi/mediasoup；边车型只接系统音频                            |
| Audio Ingest    | 麦克风、系统音频、远端音频 track、虚拟声卡 | Rust/cpal、WebRTC track subscriber、WASAPI/CoreAudio/PulseAudio    |
| Audio DSP       | 重采样、VAD、降噪、回声消除、音量归一化    | Silero VAD、RNNoise/DeepFilterNet、WebRTC AEC                      |
| ASR/SAR         | 转写、语言检测、时间戳、说话人标注        | Whisper/faster-whisper/whisper.cpp、Parakeet、sherpa-onnx、pyannote |
| T2T             | 文本翻译、术语表、风格、上下文记忆        | OPUS-MT/Marian + CTranslate2、Argos、NLLB、轻量 LLM post-edit         |
| TTS             | 目标语言合成、说话人音色、流式输出        | NeuTTS、Kokoro、Piper、XTTS、GPT-SoVITS                              |
| Orchestrator    | 流水线编排、背压、失败重试、阶段切换       | Rust Tokio / Go；本地桌面可 in-process，服务端可 NATS/Redis Streams         |
| Model Registry  | 模型版本、量化格式、许可证、评测分数、运行设备  | SQLite/Postgres + 本地模型目录                                         |
| Training Studio | 数据清洗、标注、微调、评测、发布         | Python 训练环境，推理产物导出到 ONNX/CT2/GGUF                                |
| Storage         | 音频、转写、翻译、TTS、摘要、审计       | 桌面 SQLite；企业版 Postgres + 本地对象存储                                  |

### 2. 事件协议是系统核心

所有阶段都不要直接互相调用，而是通过统一事件传递：

```text
AudioFrame
  → SpeechSegment
  → TranscriptSegment
  → TranslationSegment
  → SynthesisJob
  → SynthesizedAudio
  → MeetingOutput
```

每个事件必须带这些字段：

```text
session_id
track_id / participant_id
speaker_id
sequence_id
start_ms / end_ms
source_lang / target_lang
is_partial
revision
confidence
model_id
stage_id
audio_hash
trace_id
```

这样你就能做到：实时字幕可更新、翻译可回滚、TTS 可取消、模型可 A/B 测试、崩溃后可恢复、离线高质量重跑。

## 三、ASR / SAR 阶段设计

### 推荐策略

实时会议不要一上来追求最高精度，而要做 **双路径**：

```text
实时路径：低延迟、可容忍 partial 修正
离线路径：会议结束后用更大模型重跑，生成最终版纪要
```

Meetily 文中用单 worker 保证实时 chunk 顺序，这个思路是对的；但你的系统还要加上“partial/final 分层”。实时 UI 先显示 partial，只有 VAD 结束、标点稳定、ASR 输出稳定后，才提交 final segment。

### 可选本地模型

| 场景          | 推荐模型/引擎                                                                      | 适配建议                                                          |
| ----------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------- |
| 多语言会议，含中日英  | Whisper large-v3-turbo / large-v3 / medium，经 faster-whisper 或 whisper.cpp 量化 | 12GB GPU 下优先 faster-whisper int8/fp16；CPU 用 whisper.cpp Q5/Q8 |
| 英语/欧洲语言高速转写 | Parakeet-TDT 0.6B v3                                                         | 适合 25 个欧洲语言，不适合作为中日通用 ASR                                     |
| 中文边缘部署      | sherpa-onnx Paraformer / Zipformer 系列                                        | 适合离线、嵌入式、低依赖                                                  |
| VAD         | Silero VAD                                                                   | 2MB 级模型，单 CPU 线程处理 30ms+ 音频块小于 1ms，适合常驻                       |
| 说话人分离       | pyannote.audio / sherpa-onnx diarization                                     | pyannote 精度好但更重；sherpa-onnx 更适合端侧                             |

faster-whisper 的官方 benchmark 显示，它基于 CTranslate2，比 openai/whisper 更快、占用更少，并支持 CPU/GPU 上的 8-bit 量化；示例中 large-v2 int8 在 RTX 3070 Ti 8GB 上约 2.9GB VRAM，batch int8 约 4.5GB，CPU small int8 RAM 约 1.5GB。([GitHub][3]) whisper.cpp 支持 Whisper GGML 模型整数量化，用于降低内存和磁盘占用。([GitHub][4]) NVIDIA Parakeet-TDT 0.6B v3 是 600M 参数多语言 ASR，支持 25 个欧洲语言、自动标点、大小写、词级/段级时间戳，并采用 CC-BY-4.0 许可；但语言清单不含中文、日文、韩文，所以不要把它当泛亚洲语言模型。([Hugging Face][5])

Silero VAD 的价值非常大：它轻、快、支持 8k/16kHz，训练语料覆盖 6000+ 语言，适合作为会议系统的第一道门。([GitHub][6]) 说话人分离方面，pyannote.audio 是开源 Python toolkit，提供预训练 diarization pipeline，也支持用自有数据继续 fine-tune；但要注意授权、HF token、GPU 依赖和实时性能。([GitHub][7]) sherpa-onnx 的覆盖面很广，支持本地 ASR、TTS、VAD、说话人识别/分离、语言识别、标点、增强等，并覆盖多平台和多语言 API，适合做轻量统一推理层。([GitHub][8])

## 四、T2T 翻译阶段设计

### 推荐策略

不要直接用大 LLM 做主翻译。主链路应采用 **小型 NMT 模型 + 术语表 + 可选 LLM 后编辑**：

```text
ASR final/partial text
        ↓
文本清洗：标点、数字、专有名词、说话人上下文
        ↓
术语表/翻译记忆优先替换
        ↓
NMT 翻译：OPUS-MT / Marian / NLLB / Argos
        ↓
LLM post-edit：术语一致性、商务语气、口语化
        ↓
TranslationSegment
```

这样比纯 LLM 更稳、更便宜、更容易本地部署；LLM 只做“风格和术语一致性”，而不是每个 partial 字幕都调用。

### 可选本地模型/引擎

| 场景        | 推荐方案                           | 说明                                         |
| --------- | ------------------------------ | ------------------------------------------ |
| 固定语言对、低延迟 | OPUS-MT / Marian + CTranslate2 | 优先推荐，模型小，部署稳定，适合 CPU/GPU 量化                |
| 多语言覆盖     | NLLB-200 distilled 600M        | 语言多，但许可证 CC-BY-NC，且模型卡说明主要面向研究，不建议直接做商业生产  |
| 快速离线翻译包装  | Argos Translate                | Python/CLI/GUI 方便，支持语言包和中间语言 pivot，但质量可能损失 |
| 浏览器端/纯前端  | Bergamot / Marian WASM         | 适合客户端翻译，语言覆盖有限                             |
| 术语和风格后处理  | Qwen3-4B / 3B 级本地 LLM          | 用于术语、格式、摘要、会议纪要，不建议承担主实时翻译                 |

CTranslate2 支持 INT8、FP16、BF16、INT16、4-bit AWQ 等量化方式，能显著降低模型大小并提升执行效率，适合作为本地翻译模型运行时。([OpenNMT][9]) OPUS-MT 有 1000+ 预训练模型，且基于 Marian 的 Transformer 模型可转换为 CTranslate2 格式。([OpenNMT][10]) NLLB-200 distilled 600M 支持 200 语言单句翻译，但许可证是 CC-BY-NC，模型卡还明确说主要面向研究、不是生产部署，也不适合医学/法律等特定领域长文档翻译。([Hugging Face][11]) Argos Translate 是离线翻译库，可作为 Python 库、CLI 或 GUI 使用，并通过语言包安装模型，还支持经中间语言 pivot。([GitHub][12]) Bergamot 则是本地浏览器翻译项目，强调翻译不发云端，并使用面向消费级硬件优化的机器学习模型。([browser.mt][13])

## 五、TTS 阶段设计

### 推荐策略

TTS 分三档，不要混在一起：

1. **系统默认音色**：稳定、低延迟、许可清晰，用于会议实时翻译。
2. **个人克隆音色**：需要明确授权、声纹保护、水印、使用范围控制。
3. **训练/微调音色**：只在离线训练任务中做，不能在实时会议中临时训练。

NeuTTS 很适合作为参考。它的仓库说明 NeuTTS 是本地 TTS speech language model 集合，支持 GGUF/GGML 量化、本地设备部署、3 秒音频即时 voice cloning，并且 Air/Nano 采用小 LLM backbone + 音频 codec 架构；文档还提到 streaming example 和 finetuning script。([GitHub][14]) NeuTTS Nano 是 CPU/边缘友好模型，活跃 backbone 参数约 116.8M，总参数约 228.7M，但当前英文 Nano 模型卡说明该模型是 English only，其他语言要看 multilingual collection。([Hugging Face][15]) NeuTTS Air 是约 0.5B backbone，Apache-2.0，支持 GGML，本地中端设备实时生成。([Hugging Face][16])

### 可选本地 TTS 模型

| 场景                | 推荐模型            | 说明                                                        |
| ----------------- | --------------- | --------------------------------------------------------- |
| 低延迟、许可友好          | Kokoro-82M      | 82M 参数、Apache 权重、轻量高效，适合产品默认音色                            |
| 边缘/稳定播报           | Piper           | 快速本地神经 TTS，工程成熟，适合固定音色和低资源                                |
| 本地 voice cloning  | NeuTTS Nano/Air | 本地、GGUF/GGML、3 秒克隆、可 fine-tune，注意语言覆盖                     |
| 多语言 voice cloning | XTTS-v2         | 支持 17 种语言、6 秒音频克隆、跨语言克隆，但要重点检查 Coqui Public Model License |
| 中日英/少样本微调         | GPT-SoVITS      | 支持 5 秒 zero-shot、1 分钟 few-shot fine-tune，支持英/日/韩/粤/中文     |

Kokoro 模型卡说明它是 82M 参数 open-weight TTS，Apache 权重，可用于生产和个人项目。([Hugging Face][17]) Piper 是“fast, local neural text to speech system”，适合本地稳定部署。([GitHub][18]) XTTS-v2 支持 17 种语言、6 秒音频克隆、跨语言 voice cloning 和 24kHz 输出。([Hugging Face][19]) GPT-SoVITS 支持 5 秒样本 zero-shot TTS、1 分钟 few-shot 微调，并支持英文、日文、韩文、粤语和中文。([GitHub][20])

## 六、12GB GPU / CPU 下的推荐组合

### 组合 1：最稳 MVP，本地实时字幕 + 翻译字幕

```text
VAD：Silero VAD
ASR：faster-whisper large-v3-turbo int8 / medium int8
T2T：OPUS-MT/Marian + CTranslate2 int8
TTS：先不上，或用 Piper/Kokoro 做低延迟播报
摘要：本地 3B/4B LLM 或 Ollama 小模型
```

这是我建议的第一版。先把字幕、翻译、纪要、回放和离线重跑打通，不急着做语音回灌。

### 组合 2：英语/欧洲语言会议，高速实时

```text
VAD：Silero
ASR：Parakeet-TDT 0.6B v3
T2T：OPUS-MT / NLLB fallback
TTS：NeuTTS Air/Nano 或 Kokoro
摘要：本地 LLM
```

Parakeet 的速度和时间戳能力适合英语/欧洲语言，但语言覆盖决定它不适合作为中文/日文主 ASR。([Hugging Face][5])

### 组合 3：中日英 + 可克隆语音

```text
VAD：Silero
ASR：Whisper large-v3-turbo / large-v3 int8，或 sherpa-onnx 中文模型
T2T：OPUS-MT/Marian；广覆盖 fallback 用 NLLB；术语用本地 LLM 后编辑
TTS：GPT-SoVITS / XTTS-v2 / NeuTTS 多语言模型视目标语言选择
```

这套更灵活，但 TTS 许可、音色授权、延迟和显存调度会复杂很多。

### 显存调度建议

12GB GPU 不要同时常驻所有大模型。更合理的是：

```text
GPU 优先级：
1. ASR 实时模型
2. TTS 首包生成
3. 翻译模型批处理
4. 摘要/后编辑 LLM

CPU 常驻：
VAD、轻量翻译、轻量 TTS、音频 DSP、SQLite/缓存
```

必须做一个 **VRAM Broker / Model Manager**：模型预热、空闲卸载、按阶段抢占、按语言对加载、TTS voice cache、模型版本锁定。否则 12GB 机器会出现随机 OOM、首包延迟抖动和会议中断。

## 七、可训练与自定义阶段设计

你的系统要“还能训练、支持自定义各个阶段”，重点不是在 UI 里放一个“训练按钮”，而是建立完整生命周期。

### 1. 模型注册表

每个模型和 adapter 都记录：

```text
model_id
stage: vad/asr/diarization/t2t/tts/summary
base_model
runtime: onnxruntime/ctranslate2/llama.cpp/torch/gguf
quantization: fp16/int8/q4/q5
languages
license
hardware_profile
metrics: WER/CER/BLEU/COMET/RTF/latency/MOS
created_at
approved_by
status: draft/staging/production/deprecated
```

### 2. 训练数据闭环

每次会议产生的数据不要直接拿去训练，要分层：

```text
raw audio：原始音频，只加密归档
asr correction：人工修正转写
translation correction：人工修正翻译
termbase：术语表
speaker enrollment：说话人授权样本
tts consent sample：音色授权样本
evaluation set：固定测试集，永不混进训练集
```

### 3. 每阶段怎么训练

| 阶段          | 训练方式                                              | 建议                            |
| ----------- | ------------------------------------------------- | ----------------------------- |
| ASR         | Whisper LoRA / domain fine-tune / 词表热词增强          | 先做热词和后处理，再做 LoRA；用 WER/CER 评估 |
| Diarization | pyannote fine-tune / speaker embedding enrollment | 会议室固定成员时收益明显                  |
| T2T         | 术语表、翻译记忆、OPUS/Marian 微调、LLM 后编辑 prompt/adapters   | 优先术语表和翻译记忆，微调需要足够平行语料         |
| TTS         | speaker embedding、few-shot、fine-tune、发音词典         | 必须有授权；输出加水印；不能默认克隆参会者声音       |
| Summary     | 模板、结构化 schema、轻量 LLM adapter                      | 会议纪要最好模板化，不要纯自由生成             |

Hugging Face 有 Whisper 多语言 fine-tune 指南；Coqui TTS 文档也说明它提供预训练模型、训练新模型和 fine-tune 工具。([Hugging Face][21]) NeuTTS 官方也已经提供 streaming example 和 finetuning script，因此可以纳入训练/发布流水线，但要把训练环境和会议实时运行环境隔离。([GitHub][14])

## 八、推荐的 MVP 路线

### MVP 1：本地字幕和翻译，不做语音回灌

目标是稳定：

```text
本地录音
实时 ASR
实时翻译字幕
会议结束后离线重跑
本地摘要
术语表
模型管理
```

这一版最容易做成可用产品，也最容易验证 ASR、T2T、音频链路。

### MVP 2：加入说话人和多轨

```text
每个参会者单独 track
speaker diarization / speaker id
按说话人分段翻译
点击字幕回放原音频
```

如果是自研会议系统，优先从 SFU 直接拿每个 participant 的音频 track，效果比混音后再 diarization 好很多。

### MVP 3：加入 TTS 和虚拟麦克风

```text
目标语言 TTS
虚拟麦克风输出
防回声/防自激
TTS 和原声 ducking
一键切换“字幕模式 / 译员模式 / 双语模式”
```

这一步要非常小心：TTS 输出被系统音频再次采集，会导致 ASR 识别自己的声音，然后再次翻译和播放，形成回路。必须做音源标记、AEC、输出回避和 VAD gate。

### MVP 4：训练与自定义工作流

```text
工作流编辑器
```

## 九、重点注意事项

### 1. 实时链路要接受“先错后改”

ASR partial、翻译 partial、TTS streaming 都可能修改。UI 和事件协议必须支持 revision，不要把第一版 partial 当最终事实。TTS 只应读“稳定翻译段”，不要边听 partial 边合成，否则会出现重复播报和句子断裂。

### 2. VAD 参数比模型更影响体验

Meetily 文章里强调 VAD redemption time 拉到 2000ms，避免自然停顿把长句切碎，这个判断是对的。 你的系统要按语言和会议类型动态调 VAD：日语/中文口语停顿、英语演讲、多人抢话、客服短句，参数都不同。

### 3. 不要依赖混音后的 diarization

如果能拿到每个参会者独立 track，就不要先混音。混音后的 diarization 在重叠说话、远端回声、会议室麦克风场景下会不稳定。自研 SFU 模式下，AI bot 应订阅每个 participant audio track，再做 per-speaker ASR。

### 4. TTS voice cloning 必须做授权和水印

NeuTTS、XTTS、GPT-SoVITS 这类模型让克隆非常容易，这也是产品风险。系统要有 voice consent、用途限制、可撤销、输出水印、审计日志、禁止克隆未授权参会者等机制。NeuTTS 文档本身也提到 watermarked outputs。([GitHub][14])

### 5. 许可证会决定商业可用性

NLLB-200 distilled 600M 是 CC-BY-NC，模型卡还写明主要面向研究，不应直接当商业生产翻译模型。([Hugging Face][11]) Kokoro 权重是 Apache，部署约束较少。([Hugging Face][17]) NeuTTS Air 是 Apache-2.0，但 Nano 是 NeuTTS Open License/other，XTTS-v2 是 Coqui Public Model License；这些必须逐个做 license gate。([GitHub][14])

### 6. 12GB GPU 的关键不是“能不能跑”，而是“能不能稳定并发”

单个模型能跑不代表会议可用。会议系统要同时处理音频、ASR、翻译、TTS、摘要、UI、录制、网络。建议默认让 ASR 独占 GPU，T2T 和 TTS 先走 CPU/量化，必要时短时间抢占 GPU。所有模型加载、卸载、预热都要可观测。

### 7. 要有离线重跑机制

实时链路为低延迟服务，最终纪要为准确性服务。会议结束后应自动用更高质量配置重跑：更大 ASR、完整 diarization、标点恢复、术语修正、翻译一致性检查、摘要模板化。

### 8. 评测指标必须产品化

至少要记录：

```text
ASR：WER / CER / 词级时间戳偏差 / hallucination rate
Diarization：DER / speaker confusion
T2T：BLEU / chrF / COMET / 术语命中率
TTS：RTF / 首包延迟 / MOS 主观评分 / 发音错误
系统：P50/P95 延迟 / chunk 丢失率 / OOM 次数 / 崩溃恢复成功率
```

Meetily 文章里提到 chunk 计数、丢失检测、增量 checkpoint 和崩溃恢复，这类机制要作为你的底座，而不是后期补丁。

## 最推荐的总体方案

我的建议是：

```text
第一层：会议媒体层
- 边车模式：本地桌面捕获系统音频
- 自研模式：WebRTC SFU + AI bot 订阅音频 track

第二层：本地 AI 事件总线
- AudioFrame → SpeechSegment → TranscriptSegment → TranslationSegment → SynthesisJob
- 每个事件持久化、可追踪、可回放

第三层：模型插件
- ASR：Whisper/faster-whisper/whisper.cpp + Parakeet + sherpa-onnx
- T2T：OPUS-MT/Marian/CTranslate2 + Argos + NLLB fallback + LLM post-edit
- TTS：Kokoro/Piper/NeuTTS/XTTS/GPT-SoVITS

第四层：训练与模型治理
- 数据集、adapter、量化、评测、发布、回滚、许可证 gate

第五层：产品层
- 实时字幕
- 翻译字幕
- 双语纪要
- TTS 译员模式
- 术语表
- speaker profile
- 离线高质量重跑
```

一句话概括：**以 Meetily 的本地音频工程为底座，以 NeuTTS 这类本地 TTS 为语音输出能力，但系统形态要做成“可插拔、可训练、可回放、可评测”的会议 AI 流水线，而不是写死某几个模型的桌面工具。**
