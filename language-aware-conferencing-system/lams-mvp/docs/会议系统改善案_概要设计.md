# LAMS 会议系统改善案 · 概要设计

## Context（背景与目标）

用户提供的 `会议系统架构.md` 是一份"本地会议 AI 流水线平台"的设计建议，核心三原则：**本地优先、音频链路工程化、实时链路与离线链路分离**，并给出 ASR/SAR→T2T→TTS 各阶段的模型选型、量化、训练、许可证建议。

诊断现有 LAMS 实现后发现：**系统在媒体层与流水线骨架上已经领先**——已落地文档推荐的"Mode B（WebRTC SFU + AI bot 旁路订阅）"，且已有阶段化 provider registry、双主线编排、术语表、产品化 QoS 门。真正的缺口是：**100% 云 provider（成本高）、VAD 仅能量阈值、无流式 partial 字幕、无本地模型/模型注册/训练闭环/离线高质量重跑**，以及一批已知质量缺陷（见 `docs/翻訳品質_改善点.md`）。

用户明确定位（关键澄清）：
- **Live 模式（高端实时语音）**：云 S2S，不训练、不计成本、随时用最新模型，只求**快 + 准**。
- **Lite 模式（低端实用，默认）**：本地最优小模型 LLM/NMT，**可定制、可训练**，目标**快 + 便宜 + 准**，支持 en/ja/zh/vi 够用。

本交付物 = **概要设计文档**（不动代码）。产出后写入 `lams-mvp/docs/会议系统改善案_概要设计.md`。

## 一、现状诊断（实现 vs 文档建议）

### 已达标（保留、勿重写）
| 维度 | 现状 | 对应文档建议 |
|------|------|------|
| 媒体层 | LiveKit SFU + coturn + LiveKit Agent bot；每说话人独立 track@16kHz 订阅；`agent.py` | ✅ Mode B「SFU + AI bot 订阅每 track，AI 不塞进转发路径」 |
| 双主线 | hearing(S2S 翻译音)/reading(ASR+MT 字幕)，`orchestrator.py` + `mode_router.py` | ✅「实时/离线、听/读分层」 |
| 阶段化 | ASR/MT/TTS 三槽 registry + Composite + fallback 链，env 1 行切换；`registry.py` | ✅「插件化、每阶段可替换」 |
| 背压/顺序 | 每说话人单 worker 直列、队列满弃最旧；`agent.py:192-217` | ✅「单 worker 保 chunk 顺序」 |
| 术语表 | glossary DB + prompt 注入 + 命中率计量；cache key 含 glossary 版本 | ✅「术语表优先」 |
| QoS | 分主线 P95（听≤5s/读≤4s）+ 用语命中≥95% + 数字保持≥98% + 听主线降级冷却；`qos.py` | ✅「评测指标产品化」 |
| 持久化 | TranscriptSegment + 语言别 TranslationSegment（PG）+ Redis 缓存 | ✅「音频/转写/翻译可追踪」 |
| 回声对策 | 路由级（不给说话人回自己译音 + 话者×语言分轨），非声学 | 部分（缺 AEC/防自激声学层） |

### 缺口（本案重点）
1. **成本**：全部云 provider（OpenAI/Deepgram/Google/Gemini），无本地模型 → Lite 模式成本无法压低。
2. **VAD**：`segmenter.py` 仅 RMS 能量阈值（固定 silence 600ms），非 Silero，不按语言/场景动态调 redemption。
3. **流式 partial/final**：字幕恒为 `is_final=True`（`orchestrator.py:168`）；前端 interim 通路已就绪但**后端从不发送** → Lite/Live 首字延迟体验受限。
4. **模型治理**：只有 provider registry，无"模型注册表"（版本/量化/许可证/指标/状态），模型名硬编码在 `config.py`。
5. **训练闭环**：无数据分层、无 LoRA/微调、无翻译记忆 TM、无离线高质量重跑（仅会后 minutes 总结）。
6. **说话人分离**：无 diarization（speaker_id 由房间成员外部注入）。
7. **已知质量缺陷**（`翻訳品質_改善点.md`）：噪声过滤误杀短句/数字/yes-no；mode2 未走术语/上下文/补正；语言检测 hint 锁死；`max_tokens=500` 截断；dedup 完全一致；subtitle_id 跨语言碰撞；tail flush 在 finally 外。

## 二、两模式定位与命名统一（消除混乱）

现有代码 `mode A/B` 与 quality 文档 `mode2` 命名互相矛盾。**本案统一产品命名 Lite/Live，并显式映射到内部主线**，建议后续重命名。

| 产品档位 | 内部主线（现有） | 会议 MeetingMode | 定位 | 栈 | 训练 | 成本 |
|---------|----------------|-----------------|------|-----|------|------|
| **Lite（默认）** | reading（读主线） | B | 低端实用：字幕优先 + 可选轻量译音 | **本地最优小模型** | ✅ 可定制/可训练 | 便宜 |
| **Live（可选）** | hearing（听主线） | A | 高端实时语音：S2S 翻译音 | **云 S2S 最新模型** | ❌ 不训练 | 不计 |
| （HYBRID 保留） | 双主线 | HYBRID | 同时听译音 + 读字幕 | 混合 | — | — |

- **默认 Lite**，会议室策略可切 Live（`mode_router` + participant attributes 已支持切换）。
- Lite 与 Live **代码路径不共享**（现有硬规则，保持）——Live 保持"随时换最新云模型"能力不受本地栈影响。

## 三、目标架构（五层 + 事件协议）

保留现有五层骨架，补齐事件协议与本地模型层。

```
第1层 会议媒体层      LiveKit SFU + coturn + LiveKit Agent（保留）
第2层 音频前处理      Silero VAD（替换能量VAD）+ 重采样 +（可选）RNNoise/DeepFilterNet 降噪
第3层 AI 事件总线     AudioFrame→SpeechSegment→TranscriptSegment→TranslationSegment
                      →SynthesisJob→SynthesizedAudio→MeetingOutput（typed、可持久化、可回放）
第4层 模型插件层      Lite=本地栈（faster-whisper/OPUS-MT+CT2/Kokoro-Piper）
                      Live=云 S2S（gpt-realtime/gemini-live，随时最新）
第5层 治理与产品层    模型注册表 + 训练闭环 + 离线重跑 + 字幕/译音/纪要/术语/评测
```

### 事件协议（升级重点）
现状阶段间是**直接函数调用**（依赖注入，可测），无 typed 事件、无 revision、partial 缺失。升级为统一事件，每事件带文档要求字段：
`session_id, track_id/participant_id, speaker_id, sequence_id, revision, start_ms/end_ms, source_lang/target_lang, is_partial, confidence, model_id, stage_id, audio_hash, trace_id`。

收益：**partial 字幕可更新、译音可取消、模型可 A/B、崩溃可恢复、离线可重跑**。落地上不必引入外部消息队列（进程内 Agent 足够），只需把 `_subtitle_message()` 扩为带 `revision/is_partial` 的事件、并把中间事件落 PG（现仅 final 落库）。

## 四、各阶段设计（按模式 × 阶段）

### 4.1 VAD（两模式共用）
- **替换**能量 VAD 为 **Silero VAD**（2MB、CPU 常驻、8k/16k），保留 `segmenter.py` 的可注入 `is_speech` 接口 → 最小改动换判定函数。
- **按语言/场景动态 redemption**：ja/zh 口语停顿长→silence 800–1000ms；en 演讲→短；客服短句→短。避免自然停顿切碎长句。

### 4.2 ASR
| 模式 | 方案 | 说明 |
|------|------|------|
| Lite | **faster-whisper large-v3-turbo int8**（12GB GPU）；CPU 兜底 whisper.cpp Q5 | CTranslate2 后端，支持 en/ja/zh/vi；**流式 partial**（分块 + 稳定后 final） |
| Lite（中文增强） | 可选 sherpa-onnx Paraformer/Zipformer | 中文离线端侧强 |
| Live | 云 S2S 内含 ASR（gpt-realtime）；delta transcript 作字幕 | 不单列 ASR |
- **partial/final 分层**：VAD 内先出 partial（低延迟），VAD 末尾 + 标点稳定后提交 final。接通前端已就绪的 interim 通路。
- 修复语言检测：hint 模式不锁 `language=`，改 prompt 诱导 / 或用支持 verbose_json 的模型仅做检测。

### 4.3 T2T 翻译（Lite 核心，"最优小模型"）
主链路 **小型 NMT + 术语表 + 可选 LLM 后编辑**，而非每 partial 调大 LLM：
```
ASR text → 清洗 → 术语表/翻译记忆优先替换 → NMT(OPUS-MT/Marian+CTranslate2 int8)
         → 可选 小 LLM 后编辑(术语一致/商务语气/数字校验) → TranslationSegment
```
| 模式 | 方案 |
|------|------|
| Lite 主译 | **OPUS-MT/Marian + CTranslate2 int8**（固定语言对 en/ja/zh/vi，低延迟稳定）；广覆盖兜底 NLLB（注意 CC-BY-NC 仅研究，商用需替换） |
| Lite 后编辑 | **本地 3B/4B 小 LLM（如 Qwen3-4B）**做术语一致/数字日期校验，非主译 |
| Live | 云 S2S 一体完成，不单列 MT |
- 修复 mode2：统一走带术语/上下文/补正的 `translate_text_simple`；`max_tokens` 动态化；数字/日期原文⇔译文一致校验（补正 OFF 也执行）。
- 补 **翻译记忆 TM**（现仅"会话上下文"近似）：句级 fuzzy match，跨会议复用高频译文。

### 4.4 TTS（分三档，勿混）
1. **系统默认音色**（会议实时）：Lite=**Kokoro-82M / Piper**（Apache、低延迟、本地）；Live=云 S2S 直接出译音。
2. **个人克隆音色**：需授权 + 声纹保护 + 水印 + 用途限制（NeuTTS/XTTS/GPT-SoVITS，逐个 license gate）。
3. **训练/微调音色**：仅离线训练，禁止会议中临时训练。
- Lite 译音为**可选**（字幕优先）；开启时走本地 Kokoro/Piper 压成本。

### 4.5 说话人分离（增量）
- 自研 SFU 已能拿每 participant 独立 track → **优先 per-track ASR，不做混音后 diarization**（文档强调）。
- 固定成员会议可做 speaker enrollment（embedding），收益明显。pyannote（重、精度好）/ sherpa-onnx（端侧轻）二选一，离线重跑用。

### 4.6 会议纪要（保留 + 模板化）
- 现有 `minutes.py` 会后总结保留；改为**结构化 schema + 轻量 LLM adapter**，勿纯自由生成。

## 五、训练与自定义（仅 Lite，Live 不训练）

### 5.1 模型注册表（新建，区别于 provider registry）
每模型/adapter 记录：`model_id, stage(vad/asr/diarization/t2t/tts/summary), base_model, runtime(ct2/faster-whisper/llama.cpp/onnx), quantization, languages, license, hardware_profile, metrics(WER/CER/BLEU/COMET/RTF/latency/MOS), created_at, approved_by, status(draft/staging/production/deprecated)`。模型名从 `config.py` 硬编码迁入注册表。

### 5.2 训练数据闭环（分层，勿直接拿会议数据训练）
`raw audio(加密归档) / asr correction / translation correction / termbase / speaker enrollment / tts consent / evaluation set(永不混入训练)`。

### 5.3 各阶段训练方式
| 阶段 | 方式 | 优先级 |
|------|------|--------|
| ASR | 先热词+后处理，再 Whisper LoRA/domain finetune；WER/CER 评估 | 高 |
| T2T | 术语表 + 翻译记忆优先；有平行语料再 OPUS/Marian 微调 + LLM 后编辑 prompt | 高 |
| TTS | speaker embedding/few-shot/finetune；**必须授权 + 水印**，禁默认克隆参会者 | 中 |
| Diarization | pyannote finetune / enrollment（固定成员收益大） | 低 |
| Summary | 模板 + schema + 轻量 LLM adapter | 低 |
推理产物统一导出 ONNX/CT2/GGUF，训练环境与会议实时运行环境**隔离**。

### 5.4 离线高质量重跑（新建）
会议结束自动用更高配置重跑：更大 ASR、完整 diarization、标点恢复、术语修正、翻译一致性检查、模板化纪要 → 生成最终版记录。事件协议的 `audio_hash + 持久化中间事件`是重跑前提。

## 六、优化实现要点 & 12GB 显存调度

### 6.1 VRAM Broker / Model Manager（新建，12GB 关键）
12GB 不能常驻所有大模型。GPU 优先级：`ASR 实时 > TTS 首包 > 翻译批处理 > 后编辑/摘要 LLM`；CPU 常驻：`Silero VAD、轻量翻译、Piper、音频 DSP、缓存`。功能：预热、空闲卸载、按阶段抢占、按语言对加载、TTS voice cache、版本锁定。否则会随机 OOM + 首包抖动。

### 6.2 成本优化（Lite）
本地栈默认，云仅作 Lite 兜底；同 payload 字幕合并单次 `publish_data`（多 identity，`orchestrator.py:162` 现逐人发）；翻译缓存 + TM 命中免调用；同语言跳过翻译（已有）。

### 6.3 质量缺陷修复（并入路线，来自 `翻訳品質_改善点.md`）
噪声过滤按语言化 + 数字无条件放行；mode2 走术语/上下文/补正；语言检测不锁死；`max_tokens` 动态；dedup 时间窗；subtitle_id 按语言唯一化（`{id}:{lang}`）；tail flush 移入 finally；全失败发原文/占位字幕保证"发话存在"。

## 七、QoS / 评测指标统一

统一到分主线 + 三重门（现 `HybridQoSMonitor` 已实现，废弃旧 1200ms `QoSController` 双轨并存）：
- 延迟：Live(听) P95≤5s、首包<1.5s；Lite(读) partial<500ms、final P95≤2–4s。
- 质量：术语命中≥95%、数字/日期保持≥98%、WER/CER/BLEU/COMET/DER/MOS 入注册表。
- 系统：P50/P95、chunk 丢失率、OOM 次数、崩溃恢复成功率。

## 八、分期路线（Roadmap）

- **P0 质量急修**（launch 直击）：噪声过滤、mode2 术语路径、dedup/flush/subtitle_id、失败占位。纯云、无新依赖。
- **P1 Lite 本地栈 + VAD**：Silero VAD；faster-whisper + OPUS-MT/CT2 + Kokoro 接入 registry 槽位；VRAM Broker v1。→ 成本立降。
- **P2 流式 partial/final + 事件协议**：升级事件、接通前端 interim、中间事件落库。→ 首字延迟体验。
- **P3 模型注册表 + 训练闭环**：注册表、数据分层、热词/术语/TM、LoRA、离线重跑。→ 可定制可训练。
- **P4 高级**：diarization、TTS 克隆+授权+水印、A/B、工作流插件 SDK。→ 平台化。
