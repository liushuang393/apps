# 翻訳パイプライン 改善点（品質最優先）

優先順位: **①翻訳品質 ＞ ②翻訳を漏らさない ＞ ③重複翻訳を出さない ＞ ④実時性（最後）**

3観点（セグメント化/品質LLM/プロトコル収束）で実コードを精査した結果。各項目は
**問題・根拠(file:line)・影響観点・改善案**。`★`=今回のリリース構成（mode2=`gpt4o_transcribe` + 会議室`default_mode=a`）に直撃。

---

## ① 翻訳品質

### ★Q1【最重要】mode2（mode A）が低品質MT経路を使い、用語集・文脈・補正が一切効かない
- 根拠: mode A は hearing 主線のみ駆動 → `pipeline.process_audio` → `gpt4o_transcribe.translate_audio`（`gpt4o_transcribe.py:226-301`）。
  この経路の翻訳プロンプト（`:277-300`）は **用語集なし・会話文脈なし・LLM補正なし・temperature 0.2**。
  一方、用語集/直近文脈/補正/言語別ヒントを持つ高品質MT `translate_text_simple`（`translate/routes.py`）は **reading 主線専用**で mode A では走らない。
- 影響: ① 固有名詞・専門用語の表記揺れ、主語省略の誤訳、企業用語が無視される。**品質要件に直結する最大の構造問題**。
- 改善案（推奨）: mode2 を「**ASR（transcribe）→ `translate_text_simple`（用語集/文脈/補正込み）→ TTS**」のステージ構成へ。
  既存の `registry.py`/`stages.py`（ステージ別 provider）と整合する。最小案では `translate_audio` 内のインライン翻訳を
  `translate_text_simple` 呼び出しに置換し、用語集ヒント・文脈・補正を注入する。

### ★Q2 言語検出 hint モードの `language=` 固定で誤ASR→誤訳が連鎖
- 根拠: `LANGUAGE_DETECTION_MODE=hint`（リリース既定に設定済み）→ `transcribe_with_detection`（`gpt4o_transcribe.py:138-141`）が
  話者の native_language を確定言語として `transcribe_audio(audio, hint)`、OpenAI に `language=` 固定指定（`:93-98`）。
  ja話者が英語を話すと日本語として無理に認識し音韻が近い別語に化ける。
- 背景: auto は本来正しいが `gpt-4o-transcribe` が `verbose_json` 非対応で毎回400→hint運用にした経緯。
- 影響: ①品質 ②漏れ（誤認識で意味消失）。
- 改善案: (a) 言語検出だけ `whisper-1`（verbose_json対応）で行い、ASR本体は gpt-4o-transcribe、または (b) hint でも `language=` を
  固定せず prompt 誘導に留める。会議では話者が母語以外を話す場面が普通にあるため auto 相当が望ましい。

### ★Q3 `max_tokens=500` で長い発話の訳が途中で切れる（文末欠落）
- 根拠: `gpt4o_transcribe.py:298` と `translate/routes.py:299` がともに `max_tokens=500`。長い複文発話で訳が打ち切られる。
- 影響: ①品質 ②部分漏れ。
- 改善案: 入力長に応じた動的化（例 `min(4000, len(text)*4)`）。

### Q4 翻訳キャッシュに用語集バージョンが無く、用語更新が反映されない
- 根拠: `translate/routes.py:57` の cache key は src/tgt のみ（用語集・文脈を含まない）。24h キャッシュ（`:42`）。
  `glossary.invalidate_cache()` はメモリのみで Redis 翻訳キャッシュを無効化しない。
- 影響: ①品質の遅延劣化（用語集を直しても古い訳が残る）。
- 改善案: cache key に用語集バージョン（またはハッシュ）を混ぜる。

### Q5 LLM補正が既定OFF・Gemini限定で実質未稼働
- 根拠: `llm_correction_provider="off"` 既定（`config.py:202`）、実装は Gemini のみ（`correction.py`）。補正は reading 主線のみ（`routes.py:311`）。
- 影響: ①品質（数字/固有名詞の校正が効かない）。
- 改善案: OpenAI(gpt-4o-mini)補正を実装し既定構成で有効化。最低でも**数字・日付の原文⇔訳文一致チェック**（後処理バリデーション）を補正OFFでも実行。

## ② 翻訳を漏らさない

### ★M1【最重要】ノイズフィルタが会議の正当発話を破棄
- 根拠: `base.py:167` `if len(text) <= 3: return True`（CJKの「三号」「火曜」「了解」「百万」等が全消失）。
  `NOISE_PATTERNS_EXACT`（`:43-101`）に `okay/ok/yes/no/はい/好的/是的` を完全一致で破棄。
  `MEDIA_NOISE_KEYWORDS`（`:104-`）に `ありがとう/字幕` を部分一致で破棄（「ありがとうございます」が消える）。
  呼出は `gpt4o_transcribe.py:188`、例外フォールバックでも再通過（`:200-204`）し二重に効く。
- 影響: ②漏れ（最大要因）。会議の可否確認「はい/いいえ」、数値回答、短い指示が字幕にも音声にも出ない。①数字保持要件とも衝突。
- 改善案: (a) `len<=3` を言語別化（CJKは1文字許容）、数字を含むテキストは無条件で通す。
  (b) 返答語（yes/no/はい/好的…）を除外リストから外し、純粋なフィラー（uh/um/あー）と Whisper 幻覚句（"amara.org"/"チャンネル登録" 等）だけ残す。
  (c) 「破棄」でなく「低信頼度フラグ付与で通す」（漏れ＜重複の方針に合致）。

### ★M2 dedup 完全一致が正当な繰り返しを抑制
- 根拠: `persistence.py:98-100` `is_duplicate` が room×speaker の直前1件と完全一致なら抑制。呼出 `processor.py:101`。
- 影響: ②漏れ（「はい、はい」等の正当な反復、時間的に離れた同一発話が消える）。
- 改善案: 時間窓を導入（直前と同一かつ数秒以内のみ抑制）。本来防ぐべきは「同一セグメントの二重処理」だけ。

### ★M3 tail flush が `finally` の外で、異常切断時に末尾発話が漏れる
- 根拠: `agent.py:146-152`。`async for` 終了後に `flush()`、しかし例外/キャンセルで抜けると flush 未実行（`finally` は aclose のみ）。
- 影響: ②漏れ（話し終わり直後の切断で最後の発話が消える）。
- 改善案: `flush()` を `try/finally` に移し、全離脱経路で tail を emit。

### ★M4 全主線失敗時に「発話があった事実」すら受聴者に届かない
- 根拠: `orchestrator.py:162` で `subtitle_text` 空ならメッセージ自体を出さない。mode A で hearing も縮退 reading も失敗すると無通知。
- 影響: ②漏れ（QoS warning は出るが受聴者には何も出ない）。
- 改善案: 全失敗時は原文 or「[翻訳失敗]」プレースホルダ字幕を配信し、発話の存在を保証。

### M5 SubtitleSequencer の `forget_room` が未呼出で会議再開時に冒頭発話を抑制
- 根拠: `persistence.py:106` 定義のみ・呼出ゼロ。Agent は room 常駐（`supervisor.py`）で `_last_text` が残存。
- 影響: ②漏れ（前会議の最後と同じ一言が新会議冒頭で消える）。
- 改善案: Agent 切断時に `sequencer.forget_room(room_id)`。M2の時間窓化でも自然解消。

## ③ 重複翻訳を出さない

### ★D1 subtitle_id が全言語グループで同一 → フロントが重複排除で字幕を取りこぼす
- 根拠: `processor.py:109` で1発話に1個だけ id 生成 → `orchestrate` が全 `run_group`（言語別）に同じ id を渡す（`orchestrator.py:163-168`）。
  フロント `roomStore.ts:113-119` は既出 id を破棄。
- 影響: ③重複/②漏れ（多言語受信・preference 切替直後に新言語字幕が消える）。
- 改善案: id を `f"{subtitle_id}:{target_lang}"` で言語別ユニーク化（最小修正、`_converge`）。seq は発話単位で共有のまま。

### D2 フロントの id フォールバックが不安定で二重表示/取りこぼし
- 根拠: `SubtitleDisplay.tsx:61,73` `sub.id ?? speakerId-originalText`、表示 key も idx 依存（`:180`）+ `slice(-50)`。
- 影響: ③重複・②漏れ（短文反復時）。
- 改善案: id を必須契約化（欠落メッセージは破棄）、表示 key を安定 id（D1修正後）に統一し idx を使わない。

### D3【HYBRID限定・参考】音声(S2S)と字幕(MT)で訳文が食い違い＋二重MT
- 根拠: `orchestrator.py:152` `subtitle_text = reading_text or hearing_text`。HYBRID では hearing(音声)と reading(字幕)が別API翻訳。
- 注記: **今回のリリースは mode A のため非該当**（reading が走らない）。将来 HYBRID を使う場合は字幕と音声を同一訳に一本化すること。

## ④ 実時性（最後）

### R1 字幕 data channel をリスナーごとに個別 publish（まとめ送信で軽減可）
- 根拠: `_converge` が受聴者ごとに `deliver_subtitle`（`orchestrator.py:162-176`）→ `sink._send` が1人ずつ `publish_data`。
- 改善案: 同一 payload は宛先 identity をまとめて1回送信（`destination_identities` は list 受け）。優先度低。

### R2 interim 字幕は完全な dead code（混乱源）
- 根拠: `roomStore.addInterimSubtitle`/`SubtitleDisplay` の interim 表示はあるが**送信側が存在しない**（バックエンドに interim メッセージ無し）。
- 改善案: 当面は dead code を削除。将来ストリーミングASR導入時にプロトコル（type/is_final）ごと再設計。

## 着手順の推奨（品質最優先）
1. **M1 ノイズフィルタ是正**（②漏れ最大要因・launch直撃）+ 回帰テスト（会議頻出短語のフィクスチャ）。
2. **Q1 mode2を高品質MT経路へ**（用語集/文脈/補正注入・launch直撃の品質）。
3. **M2 dedup 時間窓化** / **M3 flush finally化** / **D1 id言語別化**（漏れ・重複の確実な潰し）。
4. **Q2 言語検出** / **Q3 max_tokens動的化** / **Q4 用語集キャッシュ版** / **M4 失敗時プレースホルダ**。
5. ④実時性（R1/R2）は最後。

> 不確実点（明示）: D3 の「S2S訳 vs MT訳どちらを権威にするか」はプロダクト品質判断。本レビューは「音声と字幕を同一訳に揃える」原則のみを根拠に提案。
