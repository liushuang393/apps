# 多言語リアルタイム会議翻訳システム

## 仕様説明書・技術アーキテクチャ・設計概要

### 対象言語：日本語・中国語・英語・ベトナム語

---

## 1. 結論

本システムは、以下 2 つの翻訳パターンのみを正式対応対象とする。

| No | パターン                        | 採用候補                                            | 主用途              |
| -- | --------------------------- | ----------------------------------------------- | ---------------- |
| 1  | End-to-End Speech-to-Speech | OpenAI Realtime Translation                     | 低遅延の音声同時通訳       |
| 2  | ASR → MT + 用語集 → 字幕         | Google Cloud Speech-to-Text + Cloud Translation | 高精度字幕・議事録・企業用語対応 |

クライアント通信方式は、**WebRTC に一本化**する。
理由は、音声 uplink、翻訳音声 downlink、字幕イベント、制御イベントを同じリアルタイム基盤で扱いやすく、ブラウザ・モバイル・会議システムとの相性が最も良いからである。

LLM は以下 2 種のみを正式候補とする。

| 用途   | LLM 候補          |
| ---- | --------------- |
| 第一候補 | OpenAI GPT 系    |
| 第二候補 | Google Gemini 系 |

本システムの推奨実装は、**Google パターンを基幹モード、OpenAI パターンを低遅延同時通訳モード**として併存させる構成である。

---

# 2. 採用候補プロジェクト

## 2.1 Candidate A：OpenAI Realtime Speech-to-Speech

### 目的

OpenAI Realtime Translation を利用し、話者の音声をリアルタイムで別言語の音声へ変換する。

OpenAI の Realtime Translation は、ソース音声を専用の翻訳セッションへストリーミングし、話者が話している途中でも翻訳音声と transcript delta を受け取れる仕組みで、会議、授業、ライブ配信、ビデオルームなどの用途が想定されている。([OpenAI Developers][1])

### 適用パターン

```text
音声入力
  ↓
OpenAI Realtime Translation
  ↓
翻訳音声 + 翻訳テキスト delta
  ↓
クライアント再生 / 字幕表示
```

### 採用理由

* 低遅延の音声同時通訳 UX を作りやすい
* speech-to-speech に強い
* 翻訳音声と字幕を同時に出せる
* MVP 後の差別化機能に向く
* WebRTC との親和性が高い

### 注意点

* 企業用語の厳密制御は Google パターンより弱い
* 誤訳原因の分解が難しい
* 正式議事録のソースとしては使わない
* コストとセッション数の管理が重要
* 重要会議では必ず Google パターンの字幕・記録を併用する

---

## 2.2 Candidate B：Google ASR → MT + 用語集 → 字幕

### 目的

Google Cloud Speech-to-Text Chirp 3 で音声をテキスト化し、Google Cloud Translation と用語集・Adaptive Translation を使って高精度な翻訳字幕を生成する。

Google Chirp 3 は Speech-to-Text API V2 の最新多言語モデルで、精度・速度・話者分離・自動言語検出を強化したモデルとして説明されている。([Google Cloud Documentation][2])
Google Cloud Translation の Adaptive Translation は、双語例文データセットやリクエスト内の参考訳を使って翻訳を適応させる機能であり、ドメイン固有翻訳に向く。([Google Cloud Documentation][3])

### 適用パターン

```text
音声入力
  ↓
Google Speech-to-Text Chirp 3
  ↓
ASR transcript
  ↓
用語補正 / 正規化
  ↓
Google Cloud Translation + Glossary / Adaptive Translation
  ↓
翻訳字幕
  ↓
議事録 / 検索 / 要約
```

### 採用理由

* 会議字幕の精度改善がしやすい
* 用語集・翻訳メモリと相性が良い
* 人名、会社名、商品名、数字、契約条件を制御しやすい
* 議事録、検索、要約に展開しやすい
* エラー分析がしやすい
* MVP の主力に最適

### 注意点

* speech-to-speech より音声 UX は弱い
* ASR、翻訳、補正の各段階で遅延が発生する
* 用語集設計を間違えると逆に翻訳品質が落ちる
* partial transcript と final transcript の扱いを丁寧に設計する必要がある

# 4. クライアント通信方式

## 4.1 採用方式

クライアント通信は **WebRTC に統一**する。

WebSocket、WebTransport、独自 RTC は今回の正式設計対象から外す。
ただし、WebRTC DataChannel は WebRTC の一部として利用する。

---

## 4.2 WebRTC を選ぶ理由

WebRTC は以下を一つのリアルタイム通信基盤で扱える。

| 種類                 | 用途                             |
| ------------------ | ------------------------------ |
| Audio Track        | ユーザー音声 uplink                  |
| Remote Audio Track | 翻訳音声 downlink                  |
| DataChannel        | 字幕、ASR partial、翻訳 delta、制御イベント |
| SRTP / DTLS        | 暗号化通信                          |
| ICE / TURN         | NAT 越え                         |
| Jitter Buffer      | 音声再生安定化                        |

OpenAI の Realtime Translation でも、ブラウザアプリでは WebRTC を使って音声トラックを送信し、翻訳音声をリモート音声トラックとして受け取る構成が想定されている。([OpenAI Developers][1])

---

## 4.3 クライアント通信設計

```text
Browser / Mobile Client
  └── WebRTC PeerConnection
        ├── Local Audio Track
        │      └── マイク音声送信
        │
        ├── Remote Audio Track
        │      └── 翻訳音声受信
        │
        └── DataChannel
               ├── 原文字幕
               ├── 翻訳字幕
               ├── partial transcript
               ├── final transcript
               ├── mode change event
               ├── provider status
               └── error / fallback event
```

# 5. LLM 選定

## 5.1 LLM は 2 種のみ

本システムで利用する LLM は以下に限定する。

| LLM             | 役割                                  |
| --------------- | ----------------------------------- |
| OpenAI GPT 系    | 高品質翻訳補正、議事録、意図補完、自然な表現              |
| Google Gemini 系 | Google Cloud 連携、翻訳補正、会議要約、マルチモーダル拡張 |

Google Gemini API のモデル一覧では、stable、preview、latest、experimental などのバージョン区分でモデルが提供されている。([Google AI for Developers][4])

---

## 5.2 LLM の使いどころ

LLM はリアルタイム翻訳の主役ではなく、**補正・整形・会議理解のために使う**。

| 処理               | OpenAI GPT | Google Gemini |
| ---------------- | ---------: | ------------: |
| 翻訳後補正            |          ◎ |             ○ |
| 用語表記統一           |          ◎ |             ◎ |
| 議事録生成            |          ◎ |             ◎ |
| 決定事項抽出           |          ◎ |             ◎ |
| ToDo 抽出          |          ◎ |             ◎ |
| 会議要約             |          ◎ |             ◎ |
| Google Cloud 連携  |          ○ |             ◎ |
| speech-to-speech |          ◎ |             △ |

---

## 5.3 LLM を使わない領域

以下には LLM を直接使わない。

| 領域       | 理由                              |
| -------- | ------------------------------- |
| 生音声通信    | WebRTC が担当                      |
| ASR 第一処理 | Google Chirp 3 が担当              |
| 最低遅延音声翻訳 | OpenAI Realtime Translation が担当 |
| 用語の絶対制御  | 用語集エンジンが担当                      |
| 監査ログ     | DB とイベントログが担当                   |

### 使用条件

| 条件    | 内容            |
| ----- | ------------- |

| 会議重要度 | 中〜高           |
| 用語制御  | 高             |
| 出力優先  | 字幕            |
| 議事録用途 | 推奨            |
| 併用推奨  | OpenAI 音声同時通訳 |

# 12. 境界条件・制約条件

## 12.1 音声品質の境界条件

| 条件           | 対応             |
| ------------ | -------------- |
| 1 人 1 マイク    | 品質保証対象         |
| 複数人が 1 マイク共有 | 精度低下として扱う      |
| 同時発話         | 完全保証しない        |
| 強いノイズ        | 低信頼度表示         |
| 音声欠落         | 欠落区間を記録        |
| エコー          | クライアント側 AEC 必須 |
| 混合音声         | 話者識別精度は保証外     |

---

## 12.2 言語の境界条件

| 条件        | 対応       |
| --------- | -------- |
| 日中英越      | 正式対応     |
| 方言        | ベストエフォート |
| コードスイッチング | 検出して原文保持 |
| 固有名詞      | 用語集で補正   |
| 略語        | 事前登録が必要  |
| 専門用語      | 用語集登録が必要 |

---

## 12.3 翻訳品質の境界条件

以下は完全保証しない。

* 法的判断
* 医療判断
* 金融助言
* 曖昧な発話の正確な意図推定
* 皮肉、冗談、感情の完全再現
* 同時発話の完全分離
* 音声が欠落した部分の復元

# 17. 実装優先順位

## Phase 1：MVP

最初に作るべきもの。

```text
WebRTC 音声入力
  ↓
Google Chirp 3 ASR
  ↓
Google Translation + 用語集
  ↓
翻訳字幕
  ↓
会後 transcript / 議事録
```

Phase 1 では OpenAI speech-to-speech はまだ必須にしない。
まず字幕と議事録を完成させる。

---

## Phase 2：OpenAI Speech-to-Speech 追加

```text
WebRTC 音声入力
  ↓
OpenAI Realtime Translation
  ↓
翻訳音声 + 翻訳字幕 delta
```

ここでユーザーごとの「翻訳音声 ON/OFF」を実装する。

---

## Phase 3：ハイブリッド高品質モード

```text
同じ音声を二系統に流す

1. OpenAI：リアルタイム翻訳音声
2. Google：高精度字幕・議事録
```

この構成が最終的な最高品質モード。

# 18. 最終推奨構成

## 18.1 本番推奨

| 項目       | 採用                                     |
| -------- | -------------------------------------- |
| クライアント通信 | WebRTC                                 |
| 低遅延音声翻訳  | OpenAI Realtime Translation            |

| LLM 補正 1 | OpenAI GPT                             |
| LLM 補正 2 | Google Gemini                          |
| 議事録生成    | OpenAI GPT 優先、Gemini fallback          |
| 正式記録     | Google 字幕モードの結果を採用                     |

---

## 18.2 最終プロダクト像

```text
リアルタイムで聞く：
  OpenAI Speech-to-Speech

リアルタイムで読む：
  Google ASR → MT + 用語集 → 字幕

会後に残す：
  Google transcript + LLM 議事録

継続的に精度を上げる：
  用語集 + 修正履歴 + Golden Test Set
```

# 19. この設計で捨てるもの

今回は「最高の製品を最短で作る」ために、あえて以下を捨てる。

| 捨てるもの                  | 理由                        |
| ---------------------- | ------------------------- |
| WebSocket メイン通信        | 音声リアルタイム通信では WebRTC の方が適切 |
| WebTransport           | まだ本番会議音声の主役にする必要がない       |
| Deepgram / Azure / AWS | 候補が増えすぎるため初期対象外           |
| 多数の LLM                | 評価・運用が複雑になる               |
| OSS speech-to-speech   | 品質保証・運用負荷が高い              |
| 完全自社モデル                | 初期投資が重すぎる                 |

# 20. 提案まとめ

本提案では、多言語リアルタイム会議翻訳システムを、**OpenAI と Google の 2 系統に絞って設計する**。

**OpenAI は低遅延の音声同時通訳を担当する。**
**Google は高精度字幕、用語集、議事録、正式記録を担当する。**
**WebRTC はクライアント通信の唯一の標準方式とする。**
**LLM は OpenAI GPT と Google Gemini の 2 種に限定する。**

最終的な設計原則は以下である。

> **聞く体験は OpenAI、読む精度と残す記録は Google、考える処理は GPT/Gemini、通信は WebRTC に統一する。**

この構成が、複雑さを抑えながら、低遅延・高精度・定制能力・将来拡張性を最もバランスよく満たす。
