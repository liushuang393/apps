## 📊 攻击场景对比

### 场景 1: 重复确认支付

#### ❌ 没有保护
```
用户点击"确认支付"按钮
  ↓
网络延迟，用户再次点击
  ↓
请求1 → 确认支付，扣款 ¥1000
请求2 → 再次确认支付，扣款 ¥1000  (重复！)
  ↓
结果：用户被扣款 ¥2000 ❌
```

#### ✅ 有保护
```
用户点击"确认支付"按钮
  ↓
网络延迟，用户再次点击
  ↓
请求1 → 确认支付，扣款 ¥1000，缓存响应
请求2 → 检测到重复，返回缓存响应
  ↓
结果：用户只被扣款 ¥1000 ✅
```

### 场景 2: 恶意频繁请求

#### ❌ 没有速率限制
```
攻击者脚本：
每秒发送100次确认支付请求
  ↓
服务器处理100个请求/秒
  ↓
结果：服务器崩溃或资源耗尽 ❌
```

#### ✅ 有速率限制
```
攻击者脚本：
每秒发送100次确认支付请求
  ↓
速率限制：1分钟最多5次
  ↓
前5个请求 → 允许
后续请求 → 返回 429 Too Many Requests
  ↓
结果：服务器正常运行 ✅
```

### 场景 3: 管理员误操作退款

#### ❌ 没有保护
```
管理员误操作：
快速点击"退款"按钮3次
  ↓
请求1 → 退款 ¥1000
请求2 → 退款 ¥1000  (重复！)
请求3 → 退款 ¥1000  (重复！)
  ↓
结果：用户收到 ¥3000 退款 ❌
```

#### ✅ 有保护
```
管理员误操作：
快速点击"退款"按钮3次
  ↓
请求1 → 退款 ¥1000，缓存响应
请求2 → 检测到重复，返回缓存响应
请求3 → 检测到重复，返回缓存响应
  ↓
结果：用户只收到 ¥1000 退款 ✅
```

## 🔑 关键概念解释

### 速率限制（Rate Limiting）

```
┌─────────────────────────────────────────┐
│  用户请求                                │
└─────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│  检查 Redis 计数器                       │
│  Key: ratelimit:purchase:user123        │
│  Value: 3 (当前计数)                     │
└─────────────────────────────────────────┘
              │
       计数 < 5?
    ┌─────────┴─────────┐
    │ 是                │ 否
    ▼                   ▼
┌──────────┐      ┌──────────┐
│ 允许请求  │      │ 拒绝请求  │
│ 计数 +1   │      │ 返回 429  │
└──────────┘      └──────────┘
```

### 幂等性（Idempotency）

```
┌─────────────────────────────────────────┐
│  用户请求                                │
│  Body: { payment_intent_id: "pi_xxx" } │
└─────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│  生成唯一标识                            │
│  Key = hash(userId + path + body)       │
│  = "idempotency:user123:/confirm:abc"  │
└─────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│  检查 Redis 缓存                         │
│  Key 是否存在?                           │
└─────────────────────────────────────────┘
              │
    ┌─────────┴─────────┐
    │ 不存在             │ 存在
    ▼                    ▼
┌──────────┐      ┌──────────┐
│ 处理请求  │      │ 返回缓存  │
│ 存储响应  │      │ 响应      │
└──────────┘      └──────────┘
```

## 💡 为什么需要这些保护？

### 1. 资金安全
```
没有保护 → 重复扣款/退款 → 资金损失
有保护 → 防止重复操作 → 资金安全
```

### 2. 系统稳定性
```
没有保护 → 恶意攻击 → 服务器崩溃
有保护 → 限制请求频率 → 系统稳定
```

### 3. 用户体验
```
没有保护 → 网络重试导致重复操作 → 用户困惑
有保护 → 自动处理重复请求 → 用户体验好
```
